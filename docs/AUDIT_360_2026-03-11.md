## Aktualna Prawdziwa Architektura

### Systemy kanoniczne (aktywne):
- **Routing:** `/app/dashboard` (home), `/app/offers`, `/app/projects` (V2), `/app/clients`, `/app/calendar`, `/app/finance`
- **Shell:** `NewShellLayout` z podziałem desktop (sidebar) / mobile (bottom nav + MoreScreen)
- **Dane ofert:** tabela `offers` + `offer_items` + `offer_sends` + `offer_acceptance_links`
- **Dane projektów V2:** tabela `v2_projects` + `project_public_status_tokens`
- **Billing:** `user_subscriptions` + `subscription_events` + `stripe_events` (idempotency)
- **Auth:** Supabase Auth + `VerifyEmail.tsx` + `AuthCallback.tsx`
- **Email:** Resend API via `send-offer-email` edge function
- **AI:** OpenAI/Anthropic/Gemini via edge functions (ai-chat-agent, ai-quote-suggestions, analyze-photo, ocr-invoice, voice-quote-processor)

### Systemy legacy (nadal żywe):
- **Tabela `projects`** — używana przez: `useProjects.ts`, `Projects.tsx`, `ProjectDetail.tsx`, `QuickEstimateWorkspace.tsx`, `useAnalyticsStats.ts`, `useFinancialReports.ts`, `Calendar.tsx` (via `useProjects`)
- **`/app/jobs/*`** — redirect do `/app/projects/*` (poprawnie)
- **`offer_approvals`** — legacy flow akceptacji ofert, nadal referencowany przez `count_monthly_finalized_offers()`

### Model danych — prawda:
```
[Rejestracja] → [Weryfikacja Email] → [Login] → [Dashboard (v2_projects)]
                                                       ↓
[Nowa Oferta] → [Kreator] → [PDF] → [Wyślij email] → [Link akceptacji]
                                                       ↓
[Oferta zaakceptowana] → [Utwórz Projekt V2] → [ProjectHub]
                                                    ├── Etapy
                                                    ├── Koszty (BurnBar)
                                                    ├── Dokumenty (Dossier)
                                                    ├── Raport foto
                                                    ├── Checklist odbioru
                                                    └── Gwarancja

[Quick Estimate] → zapisuje do legacy `projects` (!) → nie do V2
```

## Top 10 Pozostałych Problemów (wg Wpływu Biznesowego)

### 1. 🔴 Billing nie działa — brak prawdziwych Stripe Price IDs
**Wpływ:** Nie ma ścieżki do monetyzacji. Żaden użytkownik nie może zapłacić.
**Akcja:** Utworzyć produkty/ceny w Stripe Dashboard, zaktualizować `plans.ts` i `PRICE_TO_PLAN_MAP`.

### 2. 🔴 Quick Estimate zapisuje do legacy `projects` zamiast `v2_projects`
**Wpływ:** Dane z Quick Estimate nie pojawiają się na dashboard ani w ProjectHub. Użytkownik traci pracę.
**Akcja:** Zmienić `QuickEstimateWorkspace.tsx` aby tworzył `offer` + opcjonalnie `v2_project`.

### 3. 🟠 Dualizm tabel `projects` / `v2_projects`
**Wpływ:** Finance, Analytics, Calendar czytają z legacy. Dashboard z V2. Użytkownik widzi niespójne dane.
**Akcja:** Migracja pozostałych hooks na `v2_projects` lub unifikacja.

### 4. 🟠 Brak upgrade journey (upsell UX)
**Wpływ:** Użytkownik trafia na blokadę AI/Voice bez zrozumienia. Nie wie jak upgrade'ować.
**Akcja:** Dodać modalne "Upgrade to Pro" z kontekstem funkcji i CTA do checkout.

### 5. 🟠 Dashboard CTA prowadzi do `/app/projects/new` zamiast `/app/offers/new`
**Wpływ:** Sprzeczne z filozofią offer-first (#386). Nowy użytkownik trafia do "secondary" flow.
**Akcja:** Zmienić nawigację na `/app/offers/new`.

### 6. 🟡 Finance/Analytics — martwe eksporty i legacy dane
**Wpływ:** Strona finance nie odzwierciedla prawdziwego stanu biznesu użytkownika.
**Akcja:** Podłączyć do v2_projects, implementować eksport lub ukryć przyciski.

### 7. 🟡 Notification URL legacy (`/app/jobs/`)
**Wpływ:** Kliknięcie notyfikacji "Oferta wysłana" prowadzi do redirect zamiast bezpośrednio.
**Akcja:** Zamienić na `/app/projects/` w `send-offer-email/index.ts:237`.

### 8. 🟡 Billing.tsx — hardcoded "2/3" usage
**Wpływ:** Fałszywa informacja. Użytkownik widzi nieprawdziwe limity.
**Akcja:** Podłączyć do prawdziwego count z v2_projects + usePlanFeatures.

### 9. 🟡 Marketplace — brak moderacji
**Wpływ:** Ryzyko spamu, fake profiles, content injection.
**Akcja:** Dodać review/report system lub ukryć do czasu implementacji.

### 10. 🟡 `useCreateSubscription` hook — dead code / security risk
**Wpływ:** Hook pozwala frontend na upsert subscription z dowolnym plan_id. RLS blokuje, ale hook nie powinien istnieć.
**Akcja:** Usunąć hook lub oznaczyć jako deprecated.
