# Majster.AI — Error Standard

> **Version:** 1.0 — foundation (PR-ERR-STD-01)
> **Source of truth:** `src/lib/errors/catalog.ts`

---

## Overview

This document describes the standardized error system for Majster.AI.
It covers: naming convention, how to add a new code, the difference between all ID types, and migration patterns.

---

## Naming Convention

```
MAJ - <DOMAIN> - <SEQ>
 │       │         │
 │       │         └─ 3-digit zero-padded sequence number (001, 002, …)
 │       └─ Short domain abbreviation (2–5 uppercase letters)
 └─ Product prefix (always MAJ for Majster.AI)
```

### Domain abbreviations

| Code prefix | Domain              |
|-------------|---------------------|
| MAJ-UNK     | Unknown / fallback  |
| MAJ-OFF     | Offers              |
| MAJ-CUS     | Customers           |
| MAJ-DOS     | Dossiers            |
| MAJ-WAR     | Warranties          |
| MAJ-CAL     | Calendar            |
| MAJ-CFG     | Configuration       |
| MAJ-DB      | Database            |
| MAJ-AUTH    | Authentication      |

## Difference Between ID Types

The system uses three different identifiers. They serve different purposes and must not be confused:

### 1. Domain Code — `MAJ-XXX-YYY`
- **What it is:** A stable, human-readable business-facing code assigned by developers.
- **Example:** `MAJ-OFF-001`
- **Same for:** Every occurrence of the same class of error.
- **Used for:** User-facing display, support lookup by category, triage by domain team.
- **Defined in:** `src/lib/errors/catalog.ts`

### 2. Request / Debug ID — `REQ-XXXXXXXX`
- **What it is:** A unique random ID generated per error event.
- **Example:** `REQ-A3F7C291`
- **Same for:** Never — each error occurrence gets a new ID.
- **Used for:** Correlating user report with a specific Sentry event ("copy your REQ-… ID for support").
- **Generated by:** `generateDebugId()` in `src/lib/errorDiagnostics.ts`

### 3. Technical Fingerprint — `ERR-XXXXXXXX`
- **What it is:** A deterministic hash of `error.name + error.message` (djb2).
- **Example:** `ERR-3B8F1CA0`
- **Same for:** Errors with the same name and message (useful for Sentry grouping).
- **Used for:** Grouping identical errors in monitoring, internal deduplication.
- **NOT shown to the user.** Only present in Sentry `extra` context.
- **Generated by:** `getErrorCode()` in `src/lib/errorDiagnostics.ts`

### Summary

| ID             | Human-readable | Stable | Unique per event | Shown to user |
|----------------|---------------|--------|-----------------|---------------|
| MAJ-XXX-YYY    | yes           | yes    | no              | **yes**       |
| REQ-XXXXXXXX   | yes           | no     | **yes**         | **yes**       |
| ERR-XXXXXXXX   | no            | yes    | no              | no            |

## Problem Details Shape (RFC 9457)

`formatError()` produces a `problem` field with this shape:

```typescript
{
  type: "https://majster.ai/errors/offer-wizard-load-failed",
  title: "MAJ-OFF-001",
  status: 500,
  detail: "Offer wizard failed to load — check offer data fetch and component tree.",
  code: "MAJ-OFF-001",
  request_id: "REQ-A3F7C291"
}
```

This is NOT currently returned by all Edge Functions.
It is a ready-to-use shape that Edge Function adapters can adopt incrementally in future PRs.

## How to Add a New Error Code

1. **Add an entry to `src/lib/errors/catalog.ts`:**

```typescript
'MAJ-PROJ-001': {
  code: 'MAJ-PROJ-001',
  domain: 'project',
  defaultHttpStatus: 500,
  retryable: true,
  ownerActionRequired: false,
  userMessageKey: 'MAJ_PROJ_001',
  developerMessage: 'Project load failed — check project query and RLS.',
  problemType: 'https://majster.ai/errors/project-load-failed',
},
```

2. **Add the user message to all three locale files:**

`src/i18n/locales/pl.json` → `errors.catalog.MAJ_PROJ_001`
`src/i18n/locales/en.json` → `errors.catalog.MAJ_PROJ_001`
`src/i18n/locales/uk.json` → `errors.catalog.MAJ_PROJ_001`

3. **Update the table in this document.**

That's it. No other files need to change for a new code.

## Migration Patterns

### Pattern 1 — ErrorBoundary (already done)

The root `ErrorBoundary` uses `formatError()` automatically.
No code change needed in feature modules for boundary-caught errors.

### Pattern 2 — toast/catch block in a frontend component

```typescript
import { formatError } from '@/lib/errors/formatError';
import { toast } from 'sonner'; // or your toast library

async function saveOffer(draft: OfferDraft) {
  try {
    await supabase.from('offers').insert(draft);
  } catch (err) {
    const fmt = formatError(err, { domainCode: 'MAJ-OFF-002' });
    toast.error(fmt.userMessage);
    // Optionally log to Sentry with enriched context:
    logError(err instanceof Error ? err : new Error(String(err)), {
      domain_code: fmt.code,
      request_id: fmt.requestId,
    });
  }
}
```

### Pattern 3 — Edge Function / API response

```typescript
import { formatError } from '../../_shared/formatError'; // future shared copy

export default async function handler(req: Request): Promise<Response> {
  try {
    // ... business logic ...
  } catch (err) {
    const fmt = formatError(err, { domainCode: 'MAJ-DB-001' });
    return new Response(JSON.stringify(fmt.problem), {
      status: fmt.problem.status,
      headers: { 'Content-Type': 'application/problem+json' },
    });
  }
}
```

> **Note:** Edge Functions run on Deno — a separate copy of the formatter will be needed there.
> The shape is already defined and stable, making this a copy-paste operation in a future PR.

## What Users See

When an error is caught by the root ErrorBoundary, the user sees:

```
Coś poszło nie tak

Wystąpił nieoczekiwany błąd. Spróbuj odświeżyć stronę.

Kod błędu:   MAJ-UNK-001
Id zgłoszenia: REQ-A3F7C291

[Spróbuj ponownie]  [Odśwież stronę]
```

Support can map the `REQ-…` ID to a Sentry event.
Support can map the `MAJ-…` code to a domain + developer hint in this document.

## Non-Goals of This Foundation

- This PR does NOT retrofit all existing `toast.error()` call sites.
- This PR does NOT convert Edge Functions to Problem Details responses.
- This PR does NOT add a monitoring dashboard.
- Future PRs can adopt `formatError()` incrementally, one domain at a time.
